HeadyOS Docs

Everything you need to understand, configure, deploy, and extend the HeadyOS platform. Start with Quick Start to get running, then dive into Architecture to understand the system deeply.

⚡ HCFullPipeline

Five-stage deterministic execution engine with checkpoint-based self-correction and agent fan-out.

🧠 Multi-Agent Supervisor

Parallel routing to specialised agents with aggregated, deterministic result merging.

∞ Sacred Geometry UX

Unified design language — gold/cyan/magenta, breathing animations, organic rounding — across every interface.

📊 Live ORS Scoring

Operational Readiness Score gates system behaviour from full parallelism (>85) down to recovery-only (<50).

🔌 MCP Protocol

Model Context Protocol integration for structured AI tool use across all registered agents and services.

📡 REST + WebSocket API

Every subsystem is reachable via a clean REST API at port 3300. WebSocket for real-time log streaming.

Quick Start

Get a HeadyOS node running locally in under five minutes.

Prerequisites

Node.js 18+, Python 3.10+, and npm 9+ are required. Docker is optional but recommended for the full container orchestration experience.

Installation

bash
# Clone the monorepo git clone https://github.com/HeadySystems/Heady.git cd Heady # Install root dependencies npm install # Set up environment variables cp .env.example .env # Edit .env — at minimum set ANTHROPIC_API_KEY # Start the manager (port 3300) npm start
Cloud Deploy
The repo includes a render.yaml blueprint. Connect the GitHub repo to Render.com and it will auto-deploy heady-manager, the Python worker, and all configured services with zero additional setup.

Verify the Installation

bash
# Health check curl localhost:3300/api/health # System status curl localhost:3300/api/system/status # Run the pipeline curl -X POST localhost:3300/api/pipeline/run

Core Concepts

The Living System Model

HeadyOS treats the entire software stack as a living organism. Every component has a lifecycle (boot → healthy → degraded → recovering → healthy), a health score, and the ability to self-correct. The system breathes — it continuously polls its own state, detects drift, and applies corrections without human intervention whenever possible.

Determinism Principle

Given the same inputs — hcfullpipeline.yaml, resource-policies.yaml, and locked dependency versions — HeadyOS always produces the same execution plan and the same task-routing decisions. Any randomness (for load balancing, sampling, etc.) is seeded with the run ID and logged, making every result fully reproducible.

Stop Rule

Stop Rule
Build aggressively when healthy; repair first when not. HeadyOS will not continue building when significant errors exist in core infrastructure, data integrity, or security. The ORS score gates this behaviour.

Sacred Geometry Design

Every HeadyOS UI follows the Sacred Geometry visual language. The design system encodes system state into visual signals: cyan glows indicate live/healthy components, gold indicates high-priority items, magenta signals warnings or active processes. Breathing animations convey the "living" nature of the system.

HCFullPipeline

The HCFullPipeline is the core execution engine. It runs all workloads in a strict five-stage dependency order, with deep self-analysis at every checkpoint.

Stage Flow

ingest
→
plan
→
execute-major-phase
→
recover
→
finalize
StageDescriptionCan Fail Gracefully?
ingestCollect and validate all inputs. Load configs, verify schemas, run initial integrity checks.Yes
planBuild the DAG, evaluate resource policies, compute cost estimates, determine concurrency level.Yes
execute-major-phaseFan out to agents via the Supervisor. Builder, Researcher, Claude Code, and Deployer run in parallel.Partial
recoverDetect failures, apply circuit-breakers, attempt automated repair. Escalates if ORS drops below 50.Yes
finalizeAggregate results, update HeadyRegistry, sync docs, emit checkpoint report.Yes
Config Source of Truth
All stage definitions, retry policies, timeouts, and dependencies live in configs/hcfullpipeline.yaml. The pipeline reads this file on every run. Do not hardcode stage behaviour in source code.

Triggering a Run

bash
# Start a pipeline run curl -X POST localhost:3300/api/pipeline/run # Check current state curl localhost:3300/api/pipeline/state # View stage DAG curl localhost:3300/api/pipeline/dag # Run history curl localhost:3300/api/pipeline/history

Agent System

HeadyOS uses a multi-agent Supervisor pattern. The Supervisor fans tasks out to specialised agents in parallel and aggregates their results deterministically.

AgentSkillsDescription
buildercode-generation, scaffoldingGenerates and assembles code artefacts, builds project structures
researchersearch, analysis, summarisationGathers context, analyses codebases, produces research reports
claude-codecode-generation, refactoring, debugging, architectureClaude-powered AI coding agent. Receives tasks from the Supervisor with full context and returns structured results
deployerdeployment, config, infrastructureHandles Render.com deployments, Docker orchestration, and environment provisioning

Direct Routing Protocol

All calls between the Supervisor and agents use direct routing — no proxy. The @heady/networking package is used with proxy: false for all internal traffic. External API calls (Anthropic, Render, etc.) go through circuit breakers with retry + exponential backoff.

Routing a Task

bash
# Check all agent statuses curl localhost:3300/api/supervisor/status # Route a task to the best-fit agent curl -X POST localhost:3300/api/supervisor/route \ -H 'Content-Type: application/json' \ -d '{"type":"build","payload":{"task":"generate auth module"}}' # Invoke Claude Code directly curl -X POST localhost:3300/api/pipeline/claude \ -d '{"prompt":"Refactor the pipeline engine for better error handling"}'

Supervisor Pattern

The hc-supervisor package implements the multi-agent Supervisor pattern. It is responsible for:

  • ▸ Capability matching — routes each task to the agent with the required skills
  • ▸ Parallel fan-out — dispatches multiple tasks concurrently within resource-policy limits
  • ▸ Result aggregation — merges agent outputs using a deterministic reduce strategy
  • ▸ Health gating — refuses to dispatch to unhealthy agents; reroutes automatically

HeadyBrain

HeadyBrain (packages/hc-brain/) is the meta-controller. It sits above the pipeline and supervisor, continuously monitoring system state and making high-level tuning decisions.

⚡ Auto-Tuning

Adjusts concurrency limits in real time based on error rates, latency, and resource utilisation.

🔒 Governance Checks

Evaluates every action against governance policies before allowing execution.

∞ Pattern Evaluation

Tracks which architectural patterns are implemented and recommends beneficial additions.

📊 ORS Computation

Computes and publishes the Operational Readiness Score that gates all system behaviour.

bash
# Brain status + readiness curl localhost:3300/api/brain/status # Auto-tune concurrency (pass current error rate) curl -X POST localhost:3300/api/brain/tune \ -d '{"errorRate":0.05}' # Run a governance check curl -X POST localhost:3300/api/brain/governance-check \ -d '{"action":"execute","actor":"builder","domain":"build"}'

Operational Readiness Score (ORS)

The ORS is a 0–100 score that HeadyBrain computes at every checkpoint. It gates how aggressively the system operates.

85 – 100
Full parallelism. New optimisations and large builds allowed.
70 – 85
Normal operation. Standard parallelism and all features active.
50 – 70
Maintenance mode. Reduced load. No new large builds.
0 – 50
Recovery mode. Repair only. Escalate to owner immediately.

Checkpoint Protocol

At every pipeline checkpoint, the system performs a ten-step deep analysis to detect drift and ensure all components are in sync.

StepAction
1Validate run state — pipeline definition + resource policies still current
2Compare config hashes — detect drift from repository state
3Re-evaluate health — find bottlenecks, errors, spend vs budget
4Check concept alignment — which patterns are active, suggest missing ones
5Apply approved patterns — gradual enablement at system boundaries
6Sync registry entries — update HeadyRegistry with new versions, endpoints, statuses
7Sync documentation — update docs that reference changed APIs/schemas/configs
8Validate notebooks — ensure Colab notebooks parse and reference correct APIs
9Check doc ownership freshness — flag overdue reviews per docs/DOC_OWNERS.yaml
10Report — emit comprehensive status with concept usage + config hashes
Documentation is a First-Class Concern
Outdated documentation is treated as a defect. When a mismatch between docs and behaviour is detected, an incident task is created automatically.
bash
# Run a checkpoint analysis curl -X POST localhost:3300/api/checkpoint/analyze \ -d '{"stage":"manual"}' # View checkpoint records curl localhost:3300/api/checkpoint/records # PowerShell checkpoint sync (full) .\scripts\checkpoint-sync.ps1 # Read-only drift detection .\scripts\checkpoint-sync.ps1 -Mode check

Security Model

HeadyOS enforces security at the API gateway layer using several complementary mechanisms.

🔑 API Key Auth

All API calls require a valid HEADY_API_KEY. Keys are validated using constant-time comparison to prevent timing attacks.

🚦 Rate Limiting

1,000 requests per 15-minute window on /api/ endpoints. Configurable per service in resource-policies.yaml.

🛡️ Helmet.js

All security headers (CSP, HSTS, X-Frame-Options, etc.) set via Helmet middleware on every response.

🔒 Least Privilege

Each agent is granted only the permissions it needs. Governance policies define allowed actions per actor per domain.

Never hardcode secrets
All secrets must be passed via environment variables. The codebase must never contain API keys, tokens, or credentials in source files. Use .env locally and Render.com's secret environment variable system in production.
