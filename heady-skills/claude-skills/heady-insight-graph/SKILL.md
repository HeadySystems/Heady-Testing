# Heady Insight Graph

Use this skill when you need to **design, build, or query the Insight Graph** — Heady's knowledge graph that discovers and surfaces connections, patterns, and emergent insights across everything a user has worked on.

## When to Use This Skill

- Designing the knowledge graph schema — nodes, edges, and properties
- Building pattern discovery — finding non-obvious connections across memories, tasks, and code
- Creating insight surfacing — proactively showing users valuable connections they missed
- Planning graph-based navigation of accumulated knowledge
- Implementing graph queries for context retrieval and exploration
- Designing the visual graph explorer interface

## Instructions

### 1. Define the Graph Schema

The Insight Graph connects all Heady data:

**Node types:**

| Type | Represents | Source |
|------|-----------|--------|
| `concept` | An idea, topic, or domain | Extracted from memories and conversations |
| `task` | A completed or active task | Task Genome |
| `file` | A source code file or document | Workspace file system |
| `person` | A person mentioned in context | Conversations, git history |
| `decision` | A choice that was made with reasoning | Memory Sanctum, task history |
| `memory` | A stored memory entry | Memory Ledger and Sanctum |
| `skill` | An installed or used skill | Skill registry |
| `insight` | A discovered pattern or connection | Generated by the graph engine |

**Edge types:**

| Edge | Meaning | Example |
|------|---------|---------|
| `relates_to` | General semantic relationship | concept → concept |
| `produced_by` | Artifact created by a task | file → task |
| `depends_on` | Dependency relationship | task → task |
| `mentions` | Entity referenced in content | memory → person |
| `decided_in` | Decision was made during this work | decision → task |
| `evolved_into` | Concept or approach changed over time | concept(v1) → concept(v2) |
| `contradicts` | Two pieces of information conflict | memory → memory |
| `supports` | Evidence supports a conclusion | memory → insight |

### 2. Build the Ingestion Pipeline

Populate the graph from Heady data sources:

```
Sources → Entity Extraction → Relationship Detection → Graph Update

1. INGEST new data (memory entry, task completion, file change, conversation)
2. EXTRACT entities: concepts, people, decisions, files mentioned
3. DETECT relationships between new entities and existing graph nodes
4. UPDATE graph with new nodes and edges
5. SCORE edge strength based on evidence count and recency
6. TRIGGER insight generation for newly connected subgraphs
```

**Entity extraction techniques:**
- Named entity recognition for people, projects, and technologies
- Topic modeling for concept extraction
- Dependency parsing for decision and reasoning chains
- Code analysis for file and function relationships

### 3. Design the Insight Engine

Discover non-obvious connections:

| Insight Type | Detection Method | Example |
|-------------|-----------------|---------|
| **Convergence** | Multiple unrelated tasks reference the same concept | "You've mentioned 'rate limiting' in 3 different projects this month" |
| **Contradiction** | Two memories or decisions conflict | "In March you said X, but in April you decided Y" |
| **Gap** | Expected knowledge is missing from the graph | "You've worked extensively on auth but have no memory about session management" |
| **Pattern** | Recurring task or decision structure | "Every time you refactor, you follow the same 5-step pattern" |
| **Decay** | Important knowledge hasn't been accessed in a long time | "Your understanding of the payment system hasn't been refreshed in 6 months" |
| **Cluster** | Group of strongly connected nodes forming a domain | "These 12 memories form a coherent body of knowledge about distributed caching" |

**Insight generation rules:**
```yaml
convergence:
  trigger: 3+ unrelated contexts reference same concept within 30 days
  confidence: count / total_contexts
  action: surface as "emerging theme"

contradiction:
  trigger: two nodes with opposing content connected to same concept
  confidence: semantic_similarity of opposing content
  action: surface as "possible contradiction — review recommended"

gap:
  trigger: concept has many outbound edges but missing expected related concepts
  confidence: expected_edges / actual_edges
  action: surface as "knowledge gap — consider investigating"
```

### 4. Design Graph Queries

Enable powerful exploration:

| Query | Returns | Use Case |
|-------|---------|----------|
| "What do I know about X?" | All nodes connected to concept X, ranked by relevance | Quick knowledge recall |
| "How is X related to Y?" | Shortest paths between two nodes | Understanding connections |
| "What changed since last week?" | New nodes and edges added in time window | Catching up after time away |
| "What are my knowledge gaps?" | Gap insights | Self-improvement |
| "Show me everything about project Z" | Subgraph centered on project node | Project context assembly |
| "What patterns do I repeat?" | Pattern insights | Workflow optimization |

### 5. Build the Graph Explorer

Visual interface for navigating the graph:

**Views:**

| View | Layout | Best For |
|------|--------|----------|
| **Galaxy** | Force-directed graph with clusters | Exploring the full knowledge landscape |
| **Timeline** | Nodes arranged chronologically | Understanding how knowledge evolved |
| **Tree** | Hierarchical from a root concept | Drilling into a specific domain |
| **Neighbors** | Single node with immediate connections | Focused exploration |

**Interactions:**
- Click node to expand its connections
- Double-click to center and zoom
- Search to find and highlight specific nodes
- Filter by node type, date range, or edge strength
- Hover to preview node content
- Right-click to pin, hide, or investigate

### 6. Plan Graph Maintenance

Keep the graph healthy:

- **Pruning** — remove weak edges (low evidence, old, never accessed)
- **Merging** — combine duplicate or near-duplicate concept nodes
- **Reweighting** — decay edge strength over time unless reinforced
- **Validation** — periodically verify that high-confidence insights still hold
- **User curation** — let users confirm, reject, or annotate graph suggestions

## Output Format

When designing Insight Graph features, produce:

1. **Graph schema** — node types, edge types, and properties
2. **Ingestion pipeline** specification
3. **Insight detection rules** with confidence scoring
4. **Query language** examples
5. **Explorer UI** wireframes
6. **Maintenance policies**

## Tips

- **The graph is a lens, not a database** — its value is in revealing connections, not storing raw data
- **Insights must be actionable** — "these things are connected" is interesting; "you should review this contradiction" is valuable
- **Edge strength matters** — a single mention is a weak edge; repeated references across contexts is strong
- **Don't over-extract** — not everything is a meaningful concept; be selective about what becomes a node
- **User trust is earned** — start with high-confidence insights; bogus suggestions erode trust quickly
- **The graph gets more valuable over time** — this is a long-term investment; early graphs are sparse, but 6-month graphs are goldmines
