# Load‑Testing Harness and Plan

This document describes a recommended approach for performing load tests on the Heady platform generated by the project builder. Load‑testing helps ensure that the system can handle real‑world traffic volumes without degrading performance or reliability.

## Objectives

* **Baseline Throughput:** Determine the maximum requests per second (RPS) the system can handle while maintaining acceptable latency.
* **Latency Targets:** Validate that median, p95, and p99 response times stay within defined SLAs under load.
* **Error Budget:** Monitor error rates (HTTP 5xx/4xx) and resource saturation thresholds.
* **Scalability:** Measure how the system scales when adding replicas or increasing resources.

## Tooling

We recommend using **[Locust](https://locust.io)** for Python‑based, distributed load testing. Locust allows you to define user behaviours as code and run them locally or in a distributed mode. Alternative tools such as **k6**, **JMeter**, or **Artillery** may also be used.

### Getting Started with Locust

1. **Install Locust:**
   ```bash
   pip install locust
   ```
2. **Write a simple test script** (`locustfile.py`):
   ```python
   from locust import HttpUser, task, between

   class WebsiteUser(HttpUser):
       wait_time = between(1, 5)

       @task
       def load_homepage(self):
           self.client.get("/")
   ```
3. **Run Locust:**
   ```bash
   locust -H http://localhost:8000
   ```
   Then open the Locust web UI at <http://localhost:8089> to configure the number of users and spawn rate.

## Test Plan

| Phase | Description | Metrics Collected |
|------|-------------|------------------|
| **Smoke test** | Run a small number of virtual users (e.g., 10) for a short period (5 minutes) to verify the system functions under load. | Latency, error rate |
| **Baseline** | Gradually ramp up to the expected peak load (e.g., 100 RPS) over 15 minutes and run for 30 minutes. | p50/p95/p99 latency, throughput, CPU/memory usage |
| **Stress test** | Increase the load beyond the expected peak (e.g., 2× the baseline) to find the breaking point. | Maximum RPS before errors, resource saturation |
| **Soak test** | Maintain steady load at baseline for several hours to observe memory leaks, resource exhaustion, and long‑term stability. | Latency drift, error rate over time |

## Integration with CI

A minimal harness can be integrated into continuous integration pipelines. Create a script (`scripts/performance/run_load_test.sh`) that runs Locust headlessly with preset user counts and spawn rates. Example:

```bash
#!/usr/bin/env bash
set -euo pipefail

# Parameters can be adjusted via environment variables
TARGET_URL=${TARGET_URL:-"http://localhost:8000"}
USERS=${USERS:-100}
SPAWN_RATE=${SPAWN_RATE:-10}
DURATION=${DURATION:-300}

locust \
  --host "${TARGET_URL}" \
  --users "${USERS}" \
  --spawn-rate "${SPAWN_RATE}" \
  --run-time "${DURATION}s" \
  --headless \
  -f "$(dirname "$0")/locustfile.py"
```

This script can be executed in CI after deployment to ensure the application meets performance criteria before promotion.

## Reporting

The load‑testing harness should output structured metrics (e.g., JSON or InfluxDB) that can be visualised via dashboards. Integrate with the observability stack defined by the project builder (Grafana + Prometheus) to compare metrics across test runs and track performance regressions.

By following this harness and test plan, teams can proactively identify bottlenecks and ensure the Heady platform meets its performance requirements.
