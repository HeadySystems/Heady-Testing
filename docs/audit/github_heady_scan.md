## 4. Wiring and Integration Concerns (Evidence-Based)

### 4.1 Security — CRITICAL (Documented)

**Source:** [`docs/SECURITY-GAP-ANALYSIS.md`](https://github.com/HeadyMe/Heady-Main/blob/main/docs/SECURITY-GAP-ANALYSIS.md)

| Area | Coverage | Severity |
|------|----------|----------|
| MCP Tool Execution Security | 30% | CRITICAL |
| MCP Gateway Zero-Trust | 20% | CRITICAL |
| Audit & Compliance | 40% | SIGNIFICANT |
| Secret Management | 50% | MODERATE |
| CI/CD Security | 85% | Moderate |
| Network/Edge (Cloudflare) | 95% | Minimal |

**Key concern:** MCP tool execution and MCP gateway zero-trust are the two most dangerous gaps. Any arbitrary tool call via HeadyMCP can bypass auth if these layers are not closed.

### 4.2 API Key Leak in Public Repo (Documented)

**Source:** [heady-docs Issue #1](https://github.com/HeadyMe/heady-docs/issues/1)

`api/api-keys-reference.md` in the public `heady-docs` repo contained masked but partially-visible key prefixes: `pplx-FvR1...`, `sk-ant-api03-...`, `ghp_49EA...`. This was addressed in Issue #1 but warrants key rotation verification for all three services (Perplexity, Anthropic, GitHub PAT).

**Action required:** Rotate all three keys immediately if not already done; audit git history for full exposure.

### 4.3 Hardcoded Neon Credentials in Colab Notebooks (Documented)

**Source:** [Commit `706bc292`](https://github.com/HeadyMe/Heady-Main/commit/706bc292), 2026-03-16: "security: strip hardcoded Neon credentials from Colab notebooks — Remove plaintext DATABASE_URL"

PostgreSQL connection strings were committed in plain text in Jupyter notebooks. Recently removed but git history retains them. **Requires BFG Repo-Cleaner or `git filter-repo` to purge from history.**

### 4.4 Merge Conflicts in README (Active)

**Source:** [HeadyMe/Heady-Main README](https://github.com/HeadyMe/Heady-Main/blob/main/README.md)

The README.md contains unresolved git merge conflict markers (`<<<<<<< HEAD` ... `>>>>>>>`) showing divergence between HEAD and commit `233933e0`. This indicates an incomplete merge or rebase during active rearchitecture. Any tool that parses README for metadata may misbehave.

### 4.5 MCP Configuration Wiring (heady-server token)

**Source:** [HeadySystems/Heady-pre-production Issue #12](https://github.com/HeadySystems/Heady-pre-production/issues/12)

`mcp_config.json` wires `heady-mcp-server` via npx with `HEADY_API_TOKEN` injected from `${HEADY_SECRET}`. Pattern: `type: local`, `command: npx`, `args: ["-y", "heady-mcp-server"]`. The `-y` flag (auto-approve) on npx with an npm package at runtime is a supply-chain risk — an attacker who compromises the `heady-mcp-server` npm package could execute arbitrary code.

### 4.6 Cross-Domain Auth Relay (9 Domains)

**Source:** [Issue #7](https://github.com/HeadyMe/Heady-Main/issues/7), [heady-docs Issue #3](https://github.com/HeadyMe/heady-docs/issues/3)

The new `SessionManager` in `@heady/persistence` performs cross-domain auth relay for all 9 domains via httpOnly cookies. The [docs issue #3](https://github.com/HeadyMe/heady-docs/issues/3) calls out the need for a **Cross-Site Auth Contract** — as of the scan date, this contract is documented as incomplete. Cross-domain cookie relay without a finalized CORS policy creates session hijacking risk across domain boundaries.

### 4.7 Heady Latent OS "Projection" Pattern

**Source:** All `*-core` repos (headymcp-core, headyme-core, headyos-core, etc.)

All 10 `*-core` repos are described as "autonomously projected from the Heady Latent OS." Each has exactly 2 commits, a single `index.js` entrypoint, no releases, no packages, and 0 stars/forks. The `site-config.json` file is present but its schema and wiring to the parent monorepo are not documented. These repos appear to be auto-generated stubs rather than independently deployable services — their relationship to the main monorepo's `src/` modules is unclear.

**Concern:** Any downstream consumer relying on `headymcp-core` or `headyme-core` as standalone packages will receive shell repos with no substantive implementation. The real implementations live in `HeadyMe/Heady-Main/src/` and `packages/`.

### 4.8 Hybrid Language Coordination (Node → Python boundary)

**Source:** CLAUDE.md, Heady-pre-production README

The system uses Node.js (heady-manager.js on port 3300) as the MCP/API gateway with Python workers (`src/heady_project/heady_conductor.py`, `process_data.py`, `consolidated_builder.py`) invoked cross-process. The boundary between Node and Python (likely via child_process or HTTP) is not explicitly documented in accessible config files, creating a risk of timeout mismatches and error propagation gaps at the Node↔Python boundary.

### 4.9 AndroidSDK in Monorepo / .gitignore Gap

**Source:** Issue #7, Heady-Main Heady-Main commits

28,000+ AndroidSDK files were accidentally tracked in git (`.gitignore` was missing `AndroidSDK/`). Fixed in v4.1.0 issue, but repo history bloat remains.

### 4.10 Incomplete Heady-pre-production Issues (WIP)

**Source:** [Issue #9](https://github.com/HeadySystems/Heady-pre-production/issues/9) (still open in archived repo)

Jules admin IDE has an open WIP issue with unchecked items: "Run final tests and validation" and "Run code review and security scan" remain incomplete. The repo is archived (read-only), so this will never close.

### 4.11 Registry Domain Mismatch

**Source:** `heady-registry.json` (v4.1.0)

The registry declares `"domain": "headyio.com"` and `"rebuilt_from_scratch": true` with memory storage at `headyio.com/api/memory` — but the live-service README and issue descriptions reference `headysystems.com` as the primary API domain. This domain mismatch between the registry and the live service endpoints is a potential routing/discovery failure point.
