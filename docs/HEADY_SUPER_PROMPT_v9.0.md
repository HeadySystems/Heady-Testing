## §30 — OPEN-ENDED EVOLUTION ZONES

These sections are intentionally underspecified. They represent frontiers where the system should evolve through execution, experimentation, and distillation rather than upfront design.

### 30.1 — Emergent Swarm Behaviors
The 17-swarm matrix was designed top-down. As the distiller accumulates execution recipes, patterns may emerge that suggest natural swarm reorganizations. The system should detect when two swarms consistently collaborate on the same task class and propose merging them, or when a single swarm handles tasks so diverse that splitting improves routing precision. **No predefined target topology** — let the data surface it.

### 30.2 — Cross-Model Cognitive Specialization
The Battle Arena currently treats all models as general-purpose competitors. As the distiller accumulates recipes showing that Claude consistently wins on architecture tasks while Groq wins on classification, the system should evolve model-task affinity scores that bias routing beyond the current `phi_weight × csl × (1/latency) × budget_factor` formula. **The routing formula itself should be learnable.** The MAPE-K loop (§25) should track model-task affinity data from AI Gateway logs and propose routing weight adjustments.

### 30.3 — Self-Evolving CSL Thresholds
The CSL gate thresholds (ψ=0.618, ψ²=0.382) are mathematically elegant but empirically untested at scale. As the system processes thousands of tasks, the distiller should track the actual distribution of scores at decision boundaries and determine whether the thresholds should adapt per-domain. **φ-derived defaults should be starting points, not gospel.**

### 30.4 — Memory Field Dynamics
The 3-tier memory architecture uses fixed TTLs and consolidation schedules. The distiller should surface patterns in what memories are recalled vs. ignored, which consolidation cycles produce useful T2 memories vs. noise, and whether the φ-scaled timing actually outperforms simpler schedules. **Measure, don't assume.**

### 30.5 — Drupal ↔ Heady Deep Integration
The Drupal 11 frontend and the Node.js backend currently communicate via REST API. As the platform matures, deeper integration patterns may emerge — Drupal Hooks triggering HeadyBee dispatches, Twig components rendering live CSL gate visualizations, or Drupal content entities serving as T2 memory storage. **The boundary between CMS and OS should blur.**

### 30.6 — DiskANN Vector Index Evolution
HNSW (pgvector, Qdrant) is the current vector index. DiskANN (available via Upstash Vector) can handle 100M+ vectors at SSD-resident scale with lower memory footprint. Trigger evaluation when T2 archive exceeds fib(20) × 1000 = 6.7M vectors. DiskANN trades query latency for memory efficiency — acceptable for cold T2/Archive tier queries. **HNSW for hot, DiskANN for cold. Let scale determine the boundary.**

### 30.7 — QStash → Upstash Workflow Migration
Redis Streams provide at-most-once delivery. QStash provides at-least-once HTTP-based delivery. Upstash Workflow Agents API builds higher-level multi-step orchestration on QStash. As pipeline complexity grows, evaluate whether Workflow's built-in state persistence and retry management can replace custom hcfullpipeline.json (88KB) orchestration. **Start with QStash for critical stages, evaluate Workflow for full pipeline replacement.**

### 30.8 — Distilled Model Promotion Pipeline
As Tier 4 model knowledge distillation (§12) produces fine-tuned local models, a formal promotion pipeline should emerge: distill → evaluate (Gamma) → canary (5% traffic) → promote (routing mesh update) → monitor (MAPE-K tracks regression). **The pipeline shape should be discovered through Tier 2 recipe distillation of successful promotions.**

### 30.9 — Multi-Region Cloud Run
Currently single-region (us-central1). When latency SLOs (§38) require it, expand to multi-region with Cosmos DB providing the globally distributed state layer. Neon branching enables per-region database replicas. **Let SLO violations determine when to expand, not premature optimization.**
