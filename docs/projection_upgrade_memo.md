## Executive Summary

Nine core service repos flagged as `thin_projection_repo: 1` in `repo_summary.json` are currently non-functional shells. Each contains exactly 7 files: a minimal Express server (`index.js`) that serves a static HTML page and a `/health` endpoint, a `package.json` with only `express` as a dependency, a `site-config.json` with marketing copy, a cosmetic `Dockerfile`, a `deploy.yml` that only runs `npm ci && npm test` (with no actual deploy step wired), and a `README.md`. None of these repos implement the domain logic described in their own feature lists.

The monorepo (`Heady-pre-production-9f2f0642`) already contains the full architectural blueprint — a Liquid Deploy workflow that knows exactly which `src/` subdirectory belongs to each vertical, 80+ service modules, a bootstrapped orchestration pattern, an autonomous engine, and a Monte Carlo projection validator. The gap is entirely a projection problem: the `liquid-deploy.yml` workflow has never been run against these shells to populate them with real code, and even when it does run, the shells need baseline production scaffolding (auth, observability, CI deployment, env validation) so they can operate autonomously after projection.

**The fix has three tiers, applied in order:**
1. **Immediately** — add baseline production scaffolding to all 9 shells so they are self-sufficient the moment real code lands.
2. **Sprint 1 (1–2 weeks)** — complete the Liquid Deploy pipeline so projections actually push source to each vertical.
3. **Per-service (2–8 weeks)** — promote each vertical from projected shell to autonomous service by implementing its declared domain logic from the monorepo source subdirectory.

## Root Cause Analysis

### 1. The Liquid Deploy pipeline is not wired end-to-end

`Heady-pre-production-9f2f0642/.github/workflows/liquid-deploy.yml` defines a complete 3-job pipeline (detect-changes → project-to-vertical → projection-summary). It correctly maps each vertical to its monorepo source path. However, the vertical repos' own `deploy.yml` files contain no actual deployment step — the GCP deploy action is present in `headyme-core` and `headysystems-core` but guarded behind an `if: ${{ secrets.GCP_PROJECT_ID != '' }}` conditional that will silently no-op if the secret is not configured. The other 7 repos have no deploy step at all.

**Result:** Even if the monorepo pushes projection commits to the verticals, nothing deploys.

### 2. The shells have no baseline that would survive a real projection

When `liquid-deploy.yml` projects `src/api/**` into `headyapi-core`, it rsyncs source files into the vertical root. A bare `index.js` with 12 lines of Express will be overwritten, but there is no `src/config/env-schema.js`, no `src/bootstrap/` directory, no `utils/logger.js` — all the modules that the projected code imports. The vertical would fail to start because its dependency tree is incomplete.

### 3. `projected: true` is never cleared

The health endpoint hardcodes `"projected": true`. There is no automation to flip this flag once a real projection lands and passes validation. Monitoring systems and the registry have no signal that a service graduated from shell to production.

### 4. Tests are stubs

All 9 repos stub `npm test` with `echo "Tests coming soon" && exit 0`. The `deploy.yml` runs this test step before deploying, creating a false green signal in CI and making it impossible to gate deployments on real quality checks.

## Upgrade Plan

### Tier 0 — Baseline production scaffolding (apply to all 9 repos now)

These changes make each shell self-sufficient and ready to receive a real projection. They take the current 12-line `index.js` to a proper service skeleton without requiring any domain logic to be written yet.

#### 0a. Add `src/config/env-schema.js` — fail-fast environment validation

```js
// src/config/env-schema.js
'use strict';
const REQUIRED = {
  production: ['NODE_ENV', 'PORT'],
  // Add service-specific vars here as projection adds them
};
function validateEnvironment({ strict = false } = {}) {
  const env = process.env.NODE_ENV || 'development';
  const missing = (REQUIRED[env] || []).filter(k => !process.env[k]);
  if (missing.length && strict) {
    throw new Error(`Missing required env vars: ${missing.join(', ')}`);
  }
  if (missing.length) {
    console.warn('[env] Missing optional vars:', missing.join(', '));
  }
}
module.exports = { validateEnvironment };
```

#### 0b. Replace `console.log` with structured logger

Add `pino` to `dependencies`. Create `utils/logger.js`:

```js
// utils/logger.js
const pino = require('pino');
module.exports = pino({ level: process.env.LOG_LEVEL || 'info' });
```

Replace all `console.log` calls in `index.js` with `logger.info(...)`.

#### 0c. Add global error middleware and graceful shutdown to `index.js`

```js
// After all routes:
app.use((err, req, res, next) => {
  logger.error({ err, path: req.path }, 'Unhandled error');
  res.status(err.status || 500).json({ error: err.message || 'Internal server error' });
});

// Graceful shutdown
const server = app.listen(PORT, () => logger.info(`${SERVICE} running on :${PORT}`));
const shutdown = (sig) => {
  logger.info({ sig }, 'Shutdown signal received');
  server.close(() => { logger.info('Server closed'); process.exit(0); });
  setTimeout(() => process.exit(1), 10_000);
};
process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT',  () => shutdown('SIGINT'));
```

#### 0d. Update `/health` to reflect real projection status

```js
app.get('/health', (req, res) => {
  const meta = (() => {
    try { return require('./.projection-meta.json'); } catch { return null; }
  })();
  res.json({
    ok: true,
    service: SERVICE_NAME,
    domain: DOMAIN,
    projected: meta === null,          // false once a real projection has landed
    projectionSha: meta?.sourceSha || null,
    projectedAt: meta?.projectedAt || null,
    uptime: process.uptime(),
    ts: new Date().toISOString(),
  });
});
```

#### 0e. Add Docker HEALTHCHECK to all 8 missing Dockerfiles

Copy the pattern already in `headyme-core/Dockerfile`:

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=10s \
    CMD node -e "const h=require('http');h.get('http://localhost:8080/health',r=>{process.exit(r.statusCode===200?0:1)}).on('error',()=>process.exit(1))"
```

#### 0f. Add `.env.example`

```
NODE_ENV=production
PORT=8080
LOG_LEVEL=info
# Service-specific vars added here as projection lands
```

#### 0g. Replace stub test with a real smoke test

Add `jest` and `supertest` to `devDependencies`. Create `tests/health.test.js`:

```js
const request = require('supertest');
const app = require('../app'); // refactor index.js to export app

test('GET /health returns 200', async () => {
  const res = await request(app).get('/health');
  expect(res.status).toBe(200);
  expect(res.body.ok).toBe(true);
});
```

Update `package.json`:
```json
"scripts": {
  "start": "node index.js",
  "test": "jest --passWithNoTests",
  "test:ci": "jest --ci --coverage"
}
```

### Tier 1 — Complete the Liquid Deploy pipeline (Sprint 1)

The monorepo already has `liquid-deploy.yml`. The following changes complete the end-to-end circuit.

#### 1a. Configure `HEADY_BOT_TOKEN` secret in all vertical repos

The `project-to-vertical` job uses `secrets.HEADY_BOT_TOKEN` to check out and push to each vertical. This PAT must be set in the GitHub org secret store and granted write access to all 9 repos.

#### 1b. Configure `GCP_PROJECT_ID` secret (and `GCP_SA_KEY` if not using Workload Identity)

All 9 `deploy.yml` files need the Cloud Run step. Use `headyme-core/deploy.yml` and `headysystems-core/deploy.yml` as the template. The remaining 7 repos (`headyapi`, `headybot`, `headybuddy`, `headyconnection`, `headyio`, `headymcp`, `headyos`) need the deploy step added:

```yaml
      - name: Deploy to Cloud Run
        if: ${{ secrets.GCP_PROJECT_ID != '' }}
        uses: google-github-actions/deploy-cloudrun@v2
        with:
          service: <repo-name>          # e.g. headyapi, headybot, etc.
          region: us-central1
          source: .
          project_id: ${{ secrets.GCP_PROJECT_ID }}
```

#### 1c. Add `quality-gates.yml` to each vertical

The monorepo has `quality-gates.yml`. Port it (or a simplified version) to each vertical so that projections from the monorepo cannot be merged without passing linting, type checking, and tests. This is the gate that prevents a broken projection from reaching production.

#### 1d. Trigger a dry-run projection

After secrets are configured, push a no-op change to `src/shared/` in the monorepo. This will exercise the `detect-changes → project-to-vertical` pipeline against all 9 repos simultaneously. Review the Monte Carlo output logs and confirm `.projection-meta.json` is written to each vertical before enabling the push step in production.

## Promotion Checklist (per vertical)

Before a vertical repo removes `"projected": true` from its `/health` response, it must satisfy:

- [ ] Domain logic from the corresponding `src/<vertical>/` directory is projected and running
- [ ] `.projection-meta.json` exists with a non-null `sourceSha`
- [ ] `npm test` runs real tests with ≥80% coverage on business logic
- [ ] `/health` returns `"projected": false` and a valid `uptime`
- [ ] Dockerfile `HEALTHCHECK` passes in CI
- [ ] `deploy.yml` successfully deploys to Cloud Run on merge to `main`
- [ ] Environment variables are documented in `.env.example`
- [ ] No `localhost` or `127.0.0.1` hardcoded in non-test source files
- [ ] Structured logger active (pino); no bare `console.log` in production paths
- [ ] Global error middleware in place
- [ ] Graceful SIGTERM/SIGINT shutdown handler in place
- [ ] At least one downstream consumer is registered in `headyapi-core` routing table
