System Overview

Heady is a distributed multi-agent AI orchestration system built on the Sacred Geometry v4.0 framework. It coordinates 8 specialized agents across 3 Google Colab Pro+ GPU runtimes to execute autonomous trading strategies, manage risk in real-time, and maintain compliance with Apex Trader Funding rules. Every constant in the system is derived from the golden ratio (φ = 1.618) or Fibonacci sequences — no magic numbers, ever.

🐝HeadyBee
Individual agent swarm behavior — scouting, foraging, waggle dance communication. Each agent runs a HeadyBee instance for distributed task discovery and execution.
🐝🐝🐝HeadySwarm
Colony-level coordination running on Bridge Builder. Handles agent registration, capability-based routing, health monitoring, self-healing, and topology optimization.
💧Liquid Nodes
Dynamically reconfigurable compute units that migrate between Colab runtimes without service interruption. Tasks flow to the GPU with the best available capacity.

Architecture

The system is organized into three tiers: the agent layer (8 Sacred Geometry agents with specialized roles), the orchestration layer (HeadyBee/HeadySwarm with capability-based routing), and the compute layer (3 Colab Pro+ runtimes with mixed GPU types). Messages flow through a Redis pub/sub bus using typed envelopes with correlation IDs and HMAC signatures.

Quick Start

Getting the Heady system running requires three steps: bootstrap the Colab runtimes, start the swarm coordinator, and register your agents. Here's how each step works.

Step 1: Bootstrap a Colab Runtime

Drop heady_colab_runtime.py into each of your 3 Colab notebooks. Each runtime gets a specialized identity based on its GPU. The bootstrap handles GPU detection, keepalive, tunneling, and checkpoint restoration automatically.

from heady_colab_runtime import RuntimeBootstrap

# On your A100 80GB runtime:
runtime = RuntimeBootstrap("cortex", port=8001)
runtime.start()
# Output: Tunnel URL, GPU state, keepalive status, checkpoint info

Step 2: Start the Swarm Coordinator

The HeadySwarm runs on the Bridge Builder agent (typically on the Cortex runtime). It manages the colony-level coordination: agent registration, task routing, health monitoring, and self-healing.

from heady_swarm import HeadySwarm
import asyncio

swarm = HeadySwarm(bus=redis_client, logger=logger)
asyncio.create_task(swarm.start())
# Swarm is now accepting agent registrations and task submissions

Step 3: Register Agents

Each agent registers with its capabilities. The swarm uses these capabilities for routing — tasks go to the agent best equipped to handle them, not the one arbitrarily ranked "highest priority." Every agent is an equal-status worker in the swarm.

await swarm.register_agent("alpha", ["market_data", "signal_generation"], "cortex", 40000)
await swarm.register_agent("risk", ["risk_assessment", "veto_authority"], "cortex", 40000)
await swarm.register_agent("execution", ["execution", "order_routing"], "synapse", 20000)
await swarm.register_agent("data", ["data_enrichment", "vector_search"], "synapse", 20000)
await swarm.register_agent("sentinel", ["monitoring", "health_check"], "reflex", 8000)
# ... register all 8 agents

Balanced Ternary Decision Logic

Every agent decision outputs one of three states. This is more nuanced than binary yes/no and enables quorum-based consensus where NEUTRAL votes don't block but also don't endorse. The three states are mathematically motivated by balanced ternary, where each trit has values {-1, 0, +1}.

+1 APPROVE
Agent endorses the action. Contributes positively to quorum score weighted by the agent's role weight.
0 NEUTRAL
Agent has no opinion or insufficient data. Contributes zero to quorum score. Does not block but does not endorse.
-1 REJECT
Agent vetoes the action. Risk Agent reject = absolute override. Compliance reject = blocks non-risk agents.

Quorum Resolution

When multiple agents vote on an action (like a trade signal), their votes are resolved through a weighted quorum system. The Risk Agent has absolute veto power — its reject overrides everything. Compliance has secondary veto. For remaining agents, votes are weighted and compared against the golden ratio threshold (PSI ≈ 0.618) for approval or rejection.

HeadyBee — Individual Agent Behavior

Each agent runs a HeadyBee instance that gives it swarm intelligence capabilities. The bee metaphor maps naturally to distributed task allocation. The behavior lifecycle is: Scout (discover tasks in your domain) → Waggle (broadcast opportunities to the colony) → Claim (volunteer for matching tasks) → Forage (execute the work) → Report (send results back).

Energy Model

Each bee has an energy level from 0.0 (fully loaded) to 1.0 (idle). Accepting a task reduces energy proportional to the task's quality score (complexity). Energy slowly recovers after task completion. The swarm coordinator factors energy into routing decisions, preferring agents that are both capable AND available — this prevents any single agent from being overloaded.

HeadySwarm — Colony-Level Coordination

The HeadySwarm class runs exclusively on the Bridge Builder agent — the Queen Bee of the colony. It manages three concurrent loops: the self-healing loop (detects and recovers from agent failures every 5 seconds), the task dispatch loop (routes tasks to the best available agent), and the heartbeat listener (processes health signals from all agents).

Self-Healing

Failure detection uses Fibonacci-derived timeouts. Agents emit heartbeats every 8 seconds. If an agent goes silent for 34 seconds, it's marked "suspected dead." After 89 seconds of silence, it's confirmed dead and its in-flight tasks are redistributed to healthy agents with matching capabilities. All constants are Fibonacci numbers — no arbitrary timeouts.

Authentication & Authorization

The Heady system uses a layered authentication model. Every inter-agent message is signed with HMAC-SHA256 for authenticity verification. External API connections (Tradovate, Rithmic, exchange feeds) use OAuth 2.0 with scoped, short-lived JWTs. The MCP Gateway enforces role-based access control, ensuring each agent can only access the tools required by its role.

Message Signing

Every AgentMessage includes a .sign(secret) method that generates HMAC-SHA256 from the source agent ID, target agent ID, correlation ID, and timestamp. The receiving agent validates this signature before processing the message, preventing spoofing and replay attacks.

Secret Management

The absolute rule: no secrets in code, ever. All API keys, JWT signing secrets, database credentials, and exchange API tokens come from environment variables or a secret manager. The config.js pattern validates required secrets at startup and throws immediately if any are missing — fail fast, fail loud.

# Required environment variables (validated at startup)
JWT_SECRET=          # Required — no default. Signs all inter-agent JWTs.
REDIS_URL=           # Required — no default. Message bus connection.
TRADOVATE_API_KEY=   # Required for live trading.
RITHMIC_API_KEY=     # Required for market data.
ALLOWED_ORIGINS=     # Comma-separated CORS whitelist. No wildcards.
