2. THE FIVE TRUST ZONES (A-E): CONSEQUENCE-BASED TAXONOMY
-------------------------------------------------------------
Zone A (Safe Text Only) — Client/UI Layer
  * Untrusted by default
  * No executable commands can originate here
  * Inputs treated as "raw signals" — sanitized, stripped, classified
  * Creates "air gap" between user prompts and system execution

Zone B (Internal Read) — Backend/API Layer
  * Read access to non-sensitive internal state
  * Cannot authorize writes or mint capabilities
  * Assembles "Request Envelope" (identity + intent + context)

Zone C (Sensitive Read / Constrained Write) — Workspace Engine + Security
  * First "trusted" boundary
  * Policy Engine operates here, evaluating against registry.json
  * Mandatory Redaction Layer processing before data leaves Zone C

Zone D (Regulated/High Impact Writes) — Providers + Approval Service
  * Auto-triggers Intent + Approval Workflow
  * Enforces Human-in-the-Loop (HITL) or multi-party consensus (quorum)
  * Actions typically digital and reversible

Zone E (Actuation / Physical Impact) — Highest Risk
  * Physical actuators (robots, drones, industrial controllers)
  * Mandates "Simulation-First" execution path
  * Sim-to-Real gate: execute in sandbox first, mint one-time token only
    if simulation passes safety bounds + human approvals secured
  * PRIMARY PATENT PROTECTION CANDIDATE


3. THE ENFORCEMENT GATE SEQUENCE
-----------------------------------
Deterministic, linearized security for EVERY operation:

  Step 1: Policy Decision
    heady.security.evaluate_policy(user, org, workspace, op)
    Returns: decision (allow/deny/require_approval), constraints, redaction_rules

  Step 2: Capability Minting
    heady.security.mint_capability()
    Cryptographically signed, scoped, time-limited, non-transferable token
    Downstream components REJECT any request without valid capability token

  Step 3: Redaction
    heady.security.apply_redaction()
    "Minimum-necessary" filtering before external model exposure
    Prevents PII/API key leakage into Model Context Windows

  Step 4: Approvals (Zone D/E)
    heady.workflow.manage_intent()
    Generates intent object, routes to approver_set
    System enters "Pending" until quorum_size met

  Step 5: Audit Append
    heady.audit.append_event()
    Immutable record via hash-chaining (Merkle trees)
    Provides forensic evidence required by SB24-205


4. PATENT PORTFOLIO — DETAILED DRAFTING GUIDANCE
---------------------------------------------------

PATENT 1: Physical Trust-Anchored Authorization (PTAA)
  Ref: 63/953,813 | Filed: 01/04/2026
  Innovation: Physical hardware device (YubiKey/HSM/biometric terminal)
    required for Zone D/E operations. Binds "digital agent" to "physical human."
  Mechanism: Challenge-response protocol — Policy Engine issues nonce to PTA
    device, device signs with hardware-protected private key, Policy Engine
    validates before minting capability.
  Differentiation: Standard MFA authenticates a session; PTAA authorizes a
    specific atomic transaction initiated by an autonomous agent.

PATENT 2: Remote-Attested Actuator Systems (RAAS)
  Ref: 63/953,871 | Filed: 01/05/2026
  Innovation: Before AI agent can actuate a device, system verifies device is
    running authorized, unmodified firmware (Confidential Computing for edge).
  Mechanism: Attestation quote (hash of boot state, kernel, running app)
    signed by hardware TPM, compared against Golden Reference Value.
  Dynamic Trust: Trust score decays over time, forcing periodic re-attestation.

PATENT 3: Safety-Biased Authorization Architecture (SBAA)
  Ref: 63/953,910 | Filed: 01/05/2026
  Innovation: Dynamic confidence threshold for authorization based on
    environmental risk factors. Simulation Loop for ambiguity resolution.
  Mechanism: Every Zone E request routed to physics-compliant simulation
    first. Divergence analysis against safety constraints. If AI confidence
    < dynamically calculated Safety Bias threshold, defaults to Safe State.

PATENT 4: Structured Context Packets (CP/SQO)
  Ref: 63/953,369 | Filed: 01/03/2026
  Innovation: Standardized data structure encapsulating data WITH governance
    rules. Rules travel with data, enforced by Supervisor Node.
  Mechanism: Packet contains Payload + Disclosure Tier + Redaction Profile.
    Supervisor inspects destination Trust Zone and auto-applies redaction
    before delivery.
