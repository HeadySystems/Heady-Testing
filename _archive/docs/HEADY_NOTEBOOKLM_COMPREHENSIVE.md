# 4. 3D Vector Memory System

The heart of Heady is its **3D-aware vector memory** — a custom spatial indexing system that maps high-dimensional embeddings into navigable 3D coordinates.

### How It Works

1. **Embedding**: Text → 384-dimensional vector via `sentence-transformers/all-MiniLM-L6-v2`
2. **3D Projection**: 384 dims split into 3 groups of 128, averaged → (x, y, z)
3. **Zone Assignment**: 3D coords mapped to 8 octant zones (0-7)
4. **Shard Storage**: Round-robin across 5 shards with auto-eviction
5. **Retrieval**: Zone-first cosine similarity search for O(n/8) speedup

### Memory Architecture

| Component | Details |
|-----------|---------|
| Embedding Model | `all-MiniLM-L6-v2` (384 dims) |
| Shards | 5 persistent JSON files |
| Max per Shard | 2,000 vectors |
| Zone Index | 8 octants (3D space) |
| Graph Layer | Hybrid RAG with entity-relationship edges |
| Persistence | Debounced auto-save to `data/` |
| Decay Model | I(m) = αFreq(m) + βe^(-γΔt) + δSurp(m) |
| Federation | Remote tier fan-out for distributed sync |

### Evolutionary Memory Scoring

The system computes **importance scores** for each memory using:

```
I(m) = α·Freq(m) + β·e^(-γ·Δt) + δ·Surp(m)

Where:
  α = 0.3  (frequency weight)
  β = 0.4  (recency weight)
  γ = 0.00001 (decay rate in ms)
  δ = 0.3  (surprise/novelty weight)
```

Memories below the decay threshold (0.15) are candidates for garbage collection. Those above the LTM threshold (0.5) are promoted to long-term memory.

### Representation Profiles

The system supports multiple coordinate representations for different consumers:

- **Cartesian** (x, y, z) — Default for internal logic
- **Spherical** (r, θ, φ) — For radial-distance queries
- **Isometric** — For 2D visualization projections

# 5. Liquid Architecture & Template Projections

Heady's **Liquid Architecture** eliminates the traditional development cycle of writing frontend code. Instead, the system dynamically generates UIs and services from vector state.

### The Projection Pipeline

```
3D Vector Space → Template Injection → HeadyBee Assembly → Dynamic UI Projection → User Device
```

1. An agent determines a user needs a specific tool (e.g., task widget)
2. The system queries vector memory for the relevant context
3. A **HeadyBee template** is dynamically configured with the context
4. The template is **injected into reality** — projected to the user's screen
5. The projection is **synchronized across all devices** via Redis Pub/Sub

### Key Properties

- **No hardcoded UI** — All interfaces are live projections of vector state
- **Self-modifying** — Change the specification vector, the UI morphs instantly
- **Cross-device sync** — Widget state persists across mobile, desktop, IDE
- **Auto-cleanup** — Stale projections are garbage collected automatically

### Outbound Projection API

```json
GET /api/vector/projection/outbound?channel=strategic&top_k=20

Response:
{
  "architecture": "3d-vector-projection-router",

  "total_vectors": 37,
  "active_zones": 8,
  "sample": [{ "id": "mem_...", "zone": 3, "representation": {...} }]
}
```

### Injectable Templates

Each template contains:

- `templateId` — Unique identifier

- `headybee` — Atomic function binding (role, zone, vectorBinding)

# 10. Continuous Embedding Pipeline

The **Continuous Embedder** is the system's always-on context pipeline — it ensures the 3D vector space is perpetually updated with the latest reality.

### Architecture: RAM-First

```
Vector Memory IS the source of truth.
Files are just PROJECTIONS of vector state.
```

The embedder does NOT scan files. It reacts to **inbound events** from the system event bus:

| Event | Handler | What Gets Embedded |
|-------|---------|-------------------|
| `user:interaction` | `onUserInteraction` | User clicks, commands, spatial movements |
| `analyst:action` | `onAnalystAction` | AI analysis results, recommendations |
| `system:action` | `onSystemAction` | Service starts, deployments, config changes |
| `telemetry:data` | `onTelemetry` | Latency, error rates, resource usage |
| `deployment:event` | `onDeployment` | Build status, deployment outcomes |
| `error:occurred` | `onError` | Errors, stack traces, recovery actions |
| `config:changed` | `onConfigChange` | YAML/JSON config mutations |
| `bee:reacted` | `onBeeReaction` | HeadyBee task completions |
| `health:checked` | `onHealthCheck` | Service health status |
| `code:changed` | `onCodeChange` | Git commits, file modifications |

### Cycle Timing (φ-derived)

| Cycle | Interval | Purpose |
|-------|----------|---------|
| Inbound Batch | φ⁵ ≈ 11.09s | Drain queue → vector memory |
| Projection Sync | φ⁷ ≈ 29.03s | Project vector state → files |
| Environment Capture | φ⁸ ≈ 46.98s | Snapshot system environment |
| Context Health | configurable | Monitor context coverage |

### Burst Handling

When the queue exceeds the burst threshold, the embedder immediately flushes to prevent data loss — maintaining the guarantee that vector memory is always current.

### SSE Transport Implementation

The Server-Sent Events transport provides real-time, streaming connectivity:

- **Endpoint**: `GET /sse` establishes a persistent connection with `text/event-stream` content type
- **Session Management**: Each connection gets a unique session ID; tool calls are POSTed to `/messages?sessionId=<id>`
- **Heartbeat**: Server sends periodic keepalive pings to maintain the connection through proxies and load balancers
- **Reconnection**: Clients automatically reconnect on drop, with session state preserved in KV storage

### OAuth 2.1 Security

Remote MCP connections are secured through a complete OAuth 2.1 implementation:

- **Authorization Endpoint**: User authenticates through Google OAuth
- **Token Exchange**: Short-lived access tokens with refresh token rotation
- **Scope Control**: Fine-grained tool access permissions per client application
- **PKCE**: Proof Key for Code Exchange prevents authorization code interception

### Tool Discovery

MCP clients discover available tools through the standard `tools/list` JSON-RPC method. Each tool includes a complete JSON Schema for its parameters, enabling IDE autocompletion and validation. The tool registry is dynamic — new tools can be added to the edge node without any client-side changes.

Every domain is rendered by the same `sacredPage()` function in the Router Worker. This function:
