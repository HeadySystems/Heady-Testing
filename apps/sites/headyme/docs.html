Heady Platform Documentation

Complete guide to understanding, configuring, and building with the Heady autonomous intelligence platform. 51+ patents. 20 AI nodes. 21 microservices.

🚀 New here? Create your account, then follow the onboarding guide to get running in under 5 minutes.

Creating Your Account

  1. Navigate to auth.headysystems.com
  2. Click "Create Account" — enter your name, email, and password (min 6 characters)
  3. Your identity is created across all Heady properties instantly — one account, full estate access
  4. After registration, you're redirected back with an active session stored in httpOnly cookies

First Login Experience

After creating your account, the onboarding flow walks you through:

  • Workspace setup — name your workspace, set timezone and preferences
  • AI provider connection — bring your own API keys for Claude, GPT-4o, Gemini, Groq, or use the Heady gateway
  • Memory initialization — your personal 3D vector space is provisioned with pgvector
  • Tool connection — install HeadyMCP in your IDE for direct AI assistance

500 Server error — check message field for details

Task Routing with CSL

When you submit a task, the gateway embeds it as a 384-dimensional vector and scores it against each node's capability profile using Continuous Semantic Logic (CSL):

// CSL AND gate — cosine similarity between task and node
const score = cslAND(taskVector, nodeCapabilityVector);

// Route to the highest-scoring node above the threshold
if (score >= CSL_THRESHOLDS.STRONG) {  // 0.82
  routeToNode(node, task);
} else if (score >= CSL_THRESHOLDS.MODERATE) {  // 0.618 (1/φ)
  routeWithFallback(node, task);
}

Memory System

Heady uses a 3D vector memory space powered by pgvector with HNSW indexing, octree spatial partitioning, and semantic drift detection. Every interaction is stored as a 384-dimensional embedding positioned in 3D space.

How It Works

  1. Embed — text is converted to a 384-dim vector using multi-provider embeddings
  2. Position — the vector is assigned an (x, y, z) coordinate via octree partitioning
  3. Index — HNSW index enables O(log n) approximate nearest neighbor search
  4. Retrieve — queries find semantically similar memories across any dimension

Operations

Operation Complexity Description
Search O(log n) HNSW approximate nearest neighbor retrieval
Insert O(log n) Add vector with automatic octree space assignment
Batch Embed O(n) Embed and insert multiple texts in one API call
Drift Detect O(1) Cosine distance between current and reference state

Drift Detection

Semantic drift is monitored continuously. When cosine distance exceeds φ⁻¹ ≈ 0.618, the system flags drift and can auto-reconfigure:

// Phi-scaled drift threshold
const DRIFT_THRESHOLD = 1 / PHI;  // ≈ 0.618
const distance = cosineDist(currentState, referenceState);
if (distance > DRIFT_THRESHOLD) {
  emitDriftEvent({ distance, threshold: DRIFT_THRESHOLD });
  // Auto-reconfigure if determinism degrades
}

Authentication

All Heady sites share a single identity domain at auth.headysystems.com. Sessions use server-issued httpOnly cookies — no tokens are ever stored in browser localStorage.

Auth Flow

  1. User clicks "Sign In" on any Heady site → redirected to auth.headysystems.com?return=ORIGIN
  2. User enters credentials → POST /api/auth/login
  3. Server validates, creates session, sets httpOnly cookie
  4. Redirect back to ?return= URL (validated against allowlist)
  5. All subsequent API calls include the cookie automatically

Cookie Policy

Attribute Value Why
httpOnly true JavaScript cannot read the cookie (XSS protection)
Secure true Only sent over HTTPS
SameSite Strict No cross-site request forgery
Path / Available to all routes
Max-Age 86400 (24h) Auto-expire after 24 hours

Cross-Site Relay

Session sync between sites uses a controlled relay pattern. The ?return= parameter is validated server-side against an approved domain allowlist:

Continuous Semantic Logic (CSL)

CSL replaces boolean if/else chains with geometric operations on high-dimensional vectors. All decision-making uses continuous cosine scoring rather than discrete branching:

  • CSL AND — cosine similarity → measures alignment between two vectors
  • CSL OR — vector superposition → combines capability vectors
  • CSL NOT — orthogonal projection → rejects matching patterns
  • CSL GATE — threshold filter → only pass if score ≥ phi-derived threshold
  • CSL XOR — difference of projections → detects exclusive patterns
  • CSL CONSENSUS — multi-vector alignment → majority agreement scoring

This approach enables 60+ provisional patents and eliminates the brittleness of traditional rule-based systems.

Phi-Math Foundation

All system constants derive from the golden ratio φ ≈ 1.618 and the Fibonacci sequence. Zero magic numbers — every threshold, cache size, batch size, retry interval, and timeout is phi-scaled: 

const PHI = 1.618033988749895;  // Golden ratio
const PSI = 0.618033988749895;  // 1/φ — used as CSL threshold

// Fibonacci-sized resources
// Cache sizes:   fib(8)=21, fib(10)=55, fib(12)=144
// Batch sizes:   fib(5)=5,  fib(7)=13,  fib(9)=34
// Retry backoff: fib(3)=2s, fib(4)=3s,  fib(5)=5s

const CSL_THRESHOLDS = {
  STRONG:   0.82,    // High-confidence routing
  MODERATE: 0.618,   // 1/φ — standard gate
  WEAK:     0.382,   // 1/φ² — permissive gate
};

Deployment

Cloud Run (Backend)

The Heady backend runs on Google Cloud Run with auto-scaling. Deploy from the repo root:

# Deploy heady-manager to Cloud Run
gcloud run deploy heady-manager \
  --source . \
  --region us-central1 \
  --allow-unauthenticated \
  --set-env-vars="NODE_ENV=production,DATABASE_URL=$DATABASE_URL"

Cloudflare Pages (Static Sites)

Each site in the Heady estate is deployed individually:

# Deploy headyme.com
npx wrangler pages deploy apps/sites/headyme \
  --project-name=headyme --branch=main

# Deploy headysystems.com
npx wrangler pages deploy apps/sites/headysystems \
  --project-name=headysystems --branch=main

CI/CD Pipeline

The auto-commit pipeline runs every 5 minutes via cron:

  1. Stage — git add -A detects all changes
  2. Commit — timestamped commit with file count
  3. Push staging — push to HeadyMe/Heady-Staging
  4. Push production — push to HeadyMe/heady-production
  5. GitHub Actions — on push, auto-deploy to Cloud Run
⚠️ Required Environment Variables DATABASE_URL — Neon Postgres connection string
ADMIN_TOKEN — Admin API authentication token
NODE_ENV — Set to production in deployed environments
CORS_ORIGINS — Comma-separated list of allowed origins
